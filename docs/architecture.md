# 아키텍처 설계

## 1. 목표와 제약

| 항목 | 결정 |
|---|---|
| 사용자 | 한국토지신탁 도시정비사업 담당 주니어 |
| 도메인 | 도시정비사업 관련 법령·판례·유권해석 |
| 최우선 가치 | **정확성 / 환각 금지** (틀린 답변은 없느니만 못함) |
| 검색 범위 | 웹 검색 차단. `corpus/`의 법 기반 자료만 사용 |
| 답변 엔진 | DeepSeek API (비용 효율) |
| 자료 형식 | 전부 md로 통일 (법령·판례·유권해석 모두) |
| 개정 반영 | 반자동 — 개정 감지 시 명근에게 알림 → 수동 교체 |
| 배포 형태 | HTML 프론트 + Cloudflare Worker 백엔드 |

## 2. 왜 이 구조인가 (설계 결정 기록)

### 2.1 LLM은 검색하지 않는다 — 검색은 Worker가 한다
DeepSeek(LLM)은 프롬프트에 넣어준 자료로 답변만 생성한다. 스스로 자료를 뒤지지
않는다. 따라서 "어떤 자료를 넣을지"는 전적으로 Worker의 검색 로직이 통제한다.
→ 정제되지 않은 정보가 답변에 섞이는 문제는 Worker 단계에서 차단한다.

### 2.2 법령도 md로 통일 (API 실시간 조회 아님)
- 이유 1: 뼈대(조문 제목)만 색인하면 조문 내 세부 규정을 검색이 놓친다.
  전문을 md로 두어야 세부까지 벡터검색 대상이 된다.
- 이유 2: 개정을 반자동으로 반영하므로 로컬 md도 며칠 내 최신이 유지된다.
- 이유 3: 모든 자료가 md면 검색 경로가 하나로 통일된다 (법령/판례/유권해석 동일).
- 결과: 국가법령정보 API는 **답변 경로에서 제외**, **개정 감지 알림 전용**으로만 사용.

### 2.3 법률 위계를 프롬프트에 반영
답변 근거의 신뢰 우선순위는 **법령 > 판례 > 유권해석**.
system-prompt에 이 순서를 명시하여, 충돌 시 상위 규범을 우선하도록 강제한다.

## 3. 컴포넌트

### 3.1 프론트엔드 — `chatbot/public/index.html`
- 채팅 UI. 질문을 Worker로 보내고 답변을 표시.
- API 키를 절대 담지 않는다 (Worker가 보관).

### 3.2 백엔드 — `chatbot/worker/`
Cloudflare Worker. 요청당 다음을 수행:
1. **검색(Retrieval)**: 질문을 임베딩 → `corpus` 벡터 인덱스에서 유사 조각 top-K 추출.
   메타데이터로 자료유형·법령명 필터 가능.
2. **조립(Augmentation)**: `system-prompt.md` + 추출 자료 + 대화 맥락 + 질문을 결합.
3. **생성(Generation)**: DeepSeek API 호출 → 답변 + 근거 반환.

### 3.3 자료 저장소 — `corpus/`
- `laws/` 법령 전문, `precedents/` 판례, `interpretations/` 유권해석.
- 각 파일은 YAML front matter 메타데이터를 갖는다 (`docs/metadata-guide.md`).

### 3.4 개정 감지 — `scripts/` (추후)
- 정기 실행(예: 일 1회). `index/법령_시행일_인덱스.md`의 시행일과
  국가법령정보 API의 최신 시행일을 비교.
- 다르면 명근에게만 이메일/텔레그램 알림. 명근이 해당 md 교체.

## 4. 벡터검색(RAG) 설계 메모

자료가 커질 것(판례·유권해석 대량 추가)에 대비해 처음부터 벡터검색 전제.

1. **청킹(Chunking)** — 의미 단위로 분할.
   - 법령: `조` 단위 (`## 제N조(제목)` 헤딩 기준).
   - 판례: 사건 단위 (필요 시 쟁점별 분할).
   - 유권해석: 질의-회신 단위.
2. **메타데이터 태깅** — 각 조각에 `자료유형/법령명/조문번호/시행일자` 부착 → 필터·인용에 사용.
3. **임베딩** — 한국어 법률 텍스트에 강한 임베딩 모델 사용(답변용 DeepSeek과 분리 가능).
4. **벡터 저장소** — 초기엔 경량(sqlite/JSON + 코사인 유사도), 확장 시 pgvector/Chroma 등.

질의 시: 질문 임베딩 → top-K 조각 추출 → system-prompt와 결합 → DeepSeek 답변.

### 초기 검색을 벡터가 아닌 어휘검색으로 시작한 이유
corpus가 아직 수백 청크로 작고, 법률 질문은 조문의 법령 용어(매도청구·관리처분계획 등)를
그대로 포함해 어휘검색 적중률이 높다. 임베딩 API·벡터DB 없이 즉시 배포·테스트 가능하고
비용이 0이다. 자료가 커지면 `retrieve.mjs`만 벡터검색으로 교체한다(§4 설계 유지).

## 6. 벡터저장소/임베딩 선정 (결정 완료 · 구현 보류)

| 항목 | 결정 | 이유 |
|---|---|---|
| 임베딩 모델 | **Cloudflare Workers AI `@cf/baai/bge-m3`** | 다국어(한국어 법률 양호), Worker 바인딩으로 직접 호출, 외부 API·키 불필요, 저비용 |
| 벡터저장소 | **Cloudflare Vectorize** | Worker 네이티브 벡터DB, 바인딩만으로 사용, 무료 티어, 별도 인프라 0 |
| 검색 방식 | **하이브리드(BM25 + 벡터)** | 법률은 조문번호·정확용어 매칭이 중요 → BM25 유지, 벡터는 동의어·문맥 질문 보완 |

**구현 시점**: 지금은 BM25-lite로 도정법 질문이 정확히 검색되어 벡터가 불필요.
아래 시점에 구현한다.
- 질문이 조문 표현과 다른 어휘로 들어와 어휘검색이 놓치는 사례가 쌓일 때
- **판례·유권해석**을 대량 추가할 때(법령과 어휘가 달라 어휘검색만으론 약함)

**구현 개요**(착수 시): ① Vectorize 인덱스 생성 → ② `scripts/build-index`에서 각 청크를
bge-m3로 임베딩해 upsert → ③ Worker에 Vectorize·AI 바인딩 추가 → ④ `retrieve.mjs`를
BM25 점수와 벡터 유사도를 합산하는 하이브리드로 확장(인터페이스 유지).
